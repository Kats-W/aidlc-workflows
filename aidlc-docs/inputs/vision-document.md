# Vision Document — au Jibun Bank AI Agent

## Executive Summary

au Jibun Bank AI Agent は、Amazon Connect を基盤とした音声・テキスト対応の生成 AI カスタマーサポートエージェントである。
au じぶん銀行の公式ウェブサイトおよび対顧向け FAQ を継続的にクローリングしてナレッジベースを自動構築・更新し、手動メンテナンスなしで自己改善サイクルを実現する。
顧客ごとの対話履歴を蓄積してパーソナライズされた応答を提供するとともに、オムニチャネル対応（音声 ↔ テキスト切り替え）と SDK を介したネイティブアプリ組み込みを実現し、週次でウェブサイト・FAQ の改善提案を管理画面へ自動提供する。

## Business Context

### Problem Statement

au じぶん銀行のカスタマーサポートには以下の課題がある。

1. **ナレッジの陳腐化**: 公式サイトや FAQ の変更がエージェントへ手動で反映されるため、タイムラグと誤情報リスクが生じる。
2. **チャネル分断**: 電話（音声）とチャット（テキスト）が別システムで管理され、セッション中のチャネル切り替えができない。
3. **パーソナライゼーション不足**: 顧客の過去履歴を参照した個別対応が行われていない。
4. **コンテンツ品質の可視化不足**: どの FAQ がわかりにくいかをデータドリブンに特定・改善する仕組みがない。

### Business Drivers

- デジタルチャネルへのシフト加速と顧客期待値の上昇
- オペレーターコスト削減と 24 時間対応ニーズ
- au グループのモバイルファースト戦略に沿ったネイティブアプリ強化
- 生成 AI 技術の成熟による実用化タイミングの到来

### Target Users and Stakeholders

| ユーザー種別 | 説明 | 主なニーズ |
|---|---|---|
| 個人顧客 | au じぶん銀行の口座保有者・申込検討者 | 銀行手続き・商品に関する即座の回答 |
| コールセンターオペレーター | 有人対応へエスカレーションするスタッフ | AI が対応できない問い合わせの引き継ぎ情報 |
| コンテンツ管理者 | FAQ・ウェブサイトを管理する行員 | 改善提案の受け取りと優先度付け |
| システム管理者 | AWS インフラを運用する IT 部門 | 監視・セキュリティ・コスト管理 |

### Business Constraints

- AWS リージョン: ap-northeast-1（東京）を優先使用
- 金融業界のセキュリティ・コンプライアンス要件（金融庁ガイドライン、個人情報保護法）を遵守
- au じぶん銀行の公式サイト以外の外部ドメインはクローリング対象外
- robots.txt および利用規約に準拠したクローリングのみ許容
- PII（個人を特定できる情報）を会話ログに保存しない設計が必須

### Success Metrics

| メトリクス | 現在 | 目標 | 測定方法 |
|---|---|---|---|
| 一次解決率（FCR） | 不明 | 70% 以上 | Amazon Connect Contact Lens |
| 平均処理時間（AHT） | 不明 | 30% 削減 | Amazon Connect レポート |
| ナレッジ鮮度 | 手動更新（週次） | 自動更新（日次） | クローラー実行ログ |
| FAQ 改善提案サイクル | なし | 週次 10 件以内 | 管理ダッシュボード |
| 顧客満足度（CSAT） | 不明 | 4.0/5.0 以上 | ポストコール/チャットアンケート |

## Project Scope

### In Scope (MVP)

1. **Amazon Connect + Amazon Lex v2 による音声・チャットボット**
   - 自然言語理解（NLU）とダイアログ管理
   - 音声（Amazon Polly による TTS）とテキスト両対応

2. **自動ナレッジベース構築・更新パイプライン**
   - au じぶん銀行公式サイト・FAQ の定期クローリング（日次）
   - Amazon Bedrock Knowledge Bases への自動インジェスト
   - Bedrock Claude による RAG（Retrieval-Augmented Generation）回答生成

3. **自己改善サイクル**
   - 会話ログ分析による未解決・低評価インタラクションの自動検出
   - Amazon Bedrock を用いたナレッジギャップ分析
   - 検出されたギャップに基づく FAQ 改善提案の自動生成（週次、最大 10 件）

4. **オムニチャネル対応**
   - Amazon Connect のオムニチャネル機能活用
   - 音声 ↔ チャット間のセッション継続（文脈保持）

5. **SDK 対応ネイティブアプリ組み込み**
   - Amazon Connect Chat SDK（iOS/Android）を活用
   - 顧客 ID に紐づいた履歴管理

6. **顧客履歴管理とパーソナライズ**
   - 顧客ごとの会話履歴を Amazon DynamoDB に蓄積
   - 過去の問い合わせ文脈を参照したパーソナライズ応答

7. **管理ダッシュボード**
   - Amazon QuickSight または Amazon CloudWatch ダッシュボード
   - 週次改善提案の表示・承認・却下インターフェース
   - 利用統計（問い合わせ件数、FCR、チャネル別割合）

### Out of Scope (MVP)

- 口座取引・残高照会などのトランザクション処理（API 連携なし）
- 多言語対応（日本語のみ）
- 他行・競合サービスのクローリング
- エージェント側の有人チャット機能（エスカレーション先の CC システムは既存を使用）
- ウェブサイトへのライブチャットウィジェット組み込み（ネイティブアプリ優先）

### Backlog (Post-MVP)

- トランザクション API 連携（残高照会、振込確認など）
- 多言語対応（英語・中国語）
- ウェブサイトライブチャット

## Key Features

### F-01: AI 音声・テキストエージェント
Amazon Connect フローと Amazon Lex v2 を統合し、インバウンド電話コールおよびチャットメッセージを処理する。Amazon Bedrock の Claude モデルを RAG エンジンとして使用し、ナレッジベースから最適な回答を生成する。

### F-02: 自動ナレッジクローリング＆更新
AWS Lambda + Amazon EventBridge（日次スケジュール）でクローラーを起動。クローリング対象は `https://www.au-jibunbank.co.jp/` 配下のページと FAQ セクション。変更差分を検出し、Amazon Bedrock Knowledge Bases の S3 データソースへ自動アップロード後、ナレッジベースの同期をトリガーする。robots.txt を遵守し、過去コンテンツとの差分管理を行う。

### F-03: 自己改善サイクル
Amazon Connect Contact Lens でコンタクト後の会話を分析。低スコア・エスカレーション・未解決ラベルが付いた会話を抽出し、Amazon Bedrock でナレッジギャップを分類。週次で EventBridge からギャップ集計・改善提案生成を実行し、DynamoDB の提案テーブルへ書き込む。

### F-04: オムニチャネルセッション継続
Amazon Connect のオムニチャネル機能（Unified Agent Experience）を使用。コンタクト ID をセッションキーとし、音声からチャット切り替え時に文脈（直近 N ターンの要約）を引き継ぐ。

### F-05: SDK ネイティブアプリ組み込み
Amazon Connect Chat SDK（iOS/Android）を提供ドキュメントとして整備。顧客 ID（au ID に基づくハッシュ値）でセッションを識別し、DynamoDB の顧客履歴テーブルと紐づける。

### F-06: 顧客履歴とパーソナライズ
コンタクト終了時に Lambda が会話サマリーを生成し DynamoDB へ保存。次回コンタクト時にサマリーをプロンプトへ注入してパーソナライズされた回答を生成。PII はコンタクト ID ベースのリンクのみ保持し、会話テキスト内の PII は Amazon Comprehend で検出・マスク処理する。

### F-07: 管理ダッシュボード
Amazon CloudWatch ダッシュボード + AWS Lambda API バックエンドで構成。改善提案一覧（優先度スコア付き）を表示し、管理者が承認・却下・保留を記録できる。承認された提案はコンテンツ管理者へ Slack/メール通知する。

## Open Questions (Pre-declared Ambiguities)

1. **クローリング頻度の最終決定**: 日次で十分か、更新頻度が高いページは時間単位にすべきか。
2. **顧客認証連携**: ネイティブアプリからのコンタクト時、au ID 認証トークンを Amazon Connect へどのように渡すか（StartChatContact API の attributes か、Lambda オーソライザーか）。
3. **エスカレーション後の継続**: 有人対応後に再度 AI へ戻ることを許容するか。
4. **音声言語モデル**: Amazon Lex v2 の日本語対応モデルと Bedrock Claude の組み合わせで十分か、あるいは別途 ASR（例: Amazon Transcribe）を介すべきか。
5. **データ保持期間**: 会話履歴を何年間保持するか（金融規制上の要件確認が必要）。
