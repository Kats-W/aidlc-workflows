# ユーザーストーリー

# au Jibun Bank AI Agent

**凡例**

- エピック: 機能単位の大きな括り（複数スプリントにまたがる）
- ストーリー: スプリントサイズ（1〜3 日で実装可能な粒度）
- 受け入れ基準: 主要フローは Given/When/Then、補足は箇条書き

---

## Epic 1: AI 音声・テキストエージェント

> **エピック概要**: 顧客が電話またはチャットで au じぶん銀行 AI エージェントに問い合わせを行い、RAG ベースの回答を受け取れる。

---

### US-1.1: 電話での AI 問い合わせ

**ペルソナ**: ユキ（個人顧客）
**ストーリー**:
> au じぶん銀行の顧客として、電話をかけると AI が日本語で応答してくれるので、窓口時間外でも銀行サービスについてすぐに質問できる。

**受け入れ基準（BDD）**:

```gherkin
Given  顧客が au じぶん銀行のサポート番号に電話をかける
When   Amazon Connect が呼び出しを受け付け、Lex v2 AI フローに接続する
Then   Polly Neural「Kazuha」の音声で日本語のウェルカムメッセージが流れる
And    顧客の発話を Lex v2 が認識し、意図（インテント）を返す

Given  顧客が「振込の手数料について教えて」と発話する
When   Lex v2 がインテントを分類し、Lambda RAG ハンドラーを呼び出す
Then   DynamoDB ベクトルストアから関連チャンクを検索し Claude で回答を生成する
And    生成された回答を Polly で音声合成して顧客に読み上げる
And    回答の出典 URL を Connect のコンタクト属性に記録する

Given  Lambda の処理に 8 秒を超えそうな場合
When   タイムアウト閾値（6 秒）を超える
Then   「少々お待ちください」の保留メッセージを再生し応答を待機する
```

**補足チェックリスト**:

- [ ] Lambda P99 レスポンスタイム ≤ 5 秒
- [ ] 音声認識失敗時は「もう一度おっしゃっていただけますか？」と再促す（最大 3 回）
- [ ] 3 回連続認識失敗でオペレーターへエスカレーション

---

### US-1.2: チャットでの AI 問い合わせ

**ペルソナ**: ユキ（個人顧客）
**ストーリー**:
> スマートフォンアプリのユーザーとして、テキストチャットで質問を送ると即座に AI が回答してくれるので、声を出せない状況でも問い合わせができる。

**受け入れ基準（BDD）**:

```gherkin
Given  顧客がアプリ内チャット画面を開く
When   Connect Chat SDK が新規チャットセッションを開始する
Then   ウェルカムメッセージが表示され入力フィールドがアクティブになる

Given  顧客が「定期預金の金利はいくらですか？」と送信する
When   Lambda RAG ハンドラーが DynamoDB を検索し Claude で回答を生成する
Then   3 秒以内にテキスト回答がチャット画面に表示される
And    回答には参照元 URL のリンクが含まれる

Given  顧客が回答に満足できず「詳しく教えて」と追加質問する
When   Lex v2 が会話コンテキストを保持したまま追加クエリを処理する
Then   前の回答を踏まえた継続的な返答が表示される
```

**補足チェックリスト**:

- [ ] テキスト P99 レスポンスタイム ≤ 3 秒
- [ ] 入力中のタイピングインジケーター（...）を表示
- [ ] 1 セッション最大 50 ターンを超えたら「お問い合わせ内容が複雑なため担当者へ接続します」と表示

---

### US-1.3: 有人オペレーターへのエスカレーション

**ペルソナ**: ユキ（顧客）、ケンジ（オペレーター）
**ストーリー**:
> 顧客として、AI が解決できない複雑な質問のときにオペレーターへ自動的につないでもらえるので、問い合わせを諦めずに済む。
> オペレーターとして、エスカレーション時に AI の会話履歴サマリーを確認できるので、顧客に再説明させずに対応できる。

**受け入れ基準（BDD）**:

```gherkin
Given  AI が「ナレッジベースに該当する情報がありません」と判定する
When   エスカレーション条件（確信度閾値 < 0.4 または顧客が「人と話したい」と発話）を満たす
Then   「担当者にお繋ぎします」とアナウンスし Connect キューへ転送する
And    コンタクト属性に直近 5 ターンの会話サマリーを JSON で添付する

Given  オペレーターが Amazon Connect CCP でコンタクトを受ける
When   コンタクトを受け付ける
Then   CCP の画面に AI 会話サマリーが表示されている（顧客の質問要旨・AI が試みた回答・未解決の理由）
```

**補足チェックリスト**:

- [ ] エスカレーション後のキュー待機中は保留音が流れる
- [ ] オペレーターが不在の場合（キュー待機 > 3 分）は折り返し電話の案内をする
- [ ] エスカレーション発生はコンタクト属性 `escalated=true` でフラグ記録

---

### US-1.4: CSAT アンケート

**ペルソナ**: ユキ（顧客）、ケンジ（オペレーター）
**ストーリー**:
> 顧客として、問い合わせ終了後に簡単な満足度アンケートに答えられるので、サービス改善に貢献できる。

**受け入れ基準（BDD）**:

```gherkin
Given  コンタクトが解決またはオペレーター対応後に終了する
When   AI エージェントへ制御が戻る
Then   「本日の対応についてご満足いただけましたか？1〜5 で評価してください」と聞く

Given  顧客が「3」と回答する
When   Lex v2 がスコアを受け取る
Then   DynamoDB にコンタクト ID・スコア・チャネルを保存する
And    「ご回答ありがとうございました」と伝えてセッションを終了する
```

**補足チェックリスト**:

- [ ] 無回答（10 秒以内に入力なし）はスコア null で記録して終了
- [ ] スコア 1〜2 の場合は追加の一言コメント（テキストチャネルのみ）を促す

## Epic 2: 自動ナレッジクローリング＆更新

> **エピック概要**: au じぶん銀行公式サイト・FAQ が週次で自動クローリングされ、変更差分のみベクトルストアに反映される。

---

### US-2.1: 週次自動クローリング実行

**ペルソナ**: タロウ（システム管理者）
**ストーリー**:
> システム管理者として、毎週日曜深夜に自動的にクローリングが実行されるので、手動操作なしで最新コンテンツがナレッジベースに反映される。

**受け入れ基準（BDD）**:

```gherkin
Given  毎週日曜 02:00 JST（EventBridge Scheduler）になる
When   クローラー Lambda が起動する
Then   robots.txt を取得し Disallow ルールを解析する
And    許可されたページのみ httpx で非同期クローリングする
And    各リクエスト間に 1〜3 秒のランダムディレイを挿入する

Given  クローリングが完了する
When   全ページの処理が終わる
Then   クローリング結果（取得件数・差分件数・スキップ件数・エラー件数）を CloudWatch Logs に JSON 形式で出力する
```

**補足チェックリスト**:

- [ ] タイムアウト: Lambda 最大実行時間 15 分（大規模サイトは Step Functions で分割）
- [ ] robots.txt 取得失敗時はクローリングを中断して CloudWatch アラームを発報
- [ ] 1 ページあたりのサイズ上限: 1MB（超過はスキップしてログ記録）

---

### US-2.2: 差分検出と S3 更新

**ペルソナ**: アキコ（コンテンツ管理者）、タロウ（IT）
**ストーリー**:
> コンテンツ管理者として、変更があったページのみが自動的に更新されるので、公式サイトを更新すれば AI の回答が自動的に最新状態になる。

**受け入れ基準（BDD）**:

```gherkin
Given  クローラーがページを取得する
When   DynamoDB の差分管理テーブルに記録された前回ハッシュと比較する
Then   ハッシュが一致するページはスキップする
And   ハッシュが変わったページのテキストを S3 へ上書き保存する
And   削除されたページは S3 から削除し差分管理テーブルのステータスを "deleted" に更新する
```

**補足チェックリスト**:

- [ ] ハッシュアルゴリズム: SHA-256（ページ本文テキストのみ対象、HTML タグ除去後）
- [ ] S3 オブジェクトキー形式: `content/{url_hash}.txt`
- [ ] 差分管理テーブルのキー: `url`（PK）, `lastModified`（SK）

---

### US-2.3: ベクトル埋め込み生成と DynamoDB 更新

**ペルソナ**: タロウ（IT）
**ストーリー**:
> システム管理者として、差分ページのベクトルが自動的に再生成・更新されるので、クローリング後すぐに最新の回答が提供できる。

**受け入れ基準（BDD）**:

```gherkin
Given  S3 に差分コンテンツが保存される
When   埋め込み Lambda が S3 イベントをトリガーに起動する
Then   テキストを約 500 トークンのチャンクに分割する（10% オーバーラップ）
And   各チャンクを Titan Embeddings v2 でベクトル化する
And   DynamoDB ベクトルテーブルへ upsert する（chunkId, embedding, text, sourceUrl, updatedAt）

Given  削除されたページのハッシュが差分管理テーブルに "deleted" で記録されている
When   埋め込み Lambda が処理する
Then   対応する DynamoDB レコードを全件削除する
```

**補足チェックリスト**:

- [ ] チャンク分割は文章境界（句点・改行）を優先
- [ ] 1 ページあたり最大チャンク数: 20（超過は最初の 20 チャンクのみ保存し警告ログ）
- [ ] Titan Embeddings v2 ベクトル次元数: 1024

---

## Epic 3: 自己改善サイクル

> **エピック概要**: 低品質コンタクトを自動検出し、ナレッジギャップを分析して週次で最大 10 件の改善提案を生成する。

---

### US-3.1: 低品質コンタクト自動検出

**ペルソナ**: アキコ（コンテンツ管理者）
**ストーリー**:
> コンテンツ管理者として、AI が対応しきれなかったコンタクトが自動的に抽出されるので、どの分野の情報が不足しているかを把握できる。

**受け入れ基準（BDD）**:

```gherkin
Given  毎週月曜 03:00 JST に分析 Lambda が起動する
When   Contact Lens の API から過去 7 日間のコンタクト分析結果を取得する
Then   以下のいずれかに該当するコンタクトを「低品質」として抽出する
    - CSAT スコア ≤ 2
    - エスカレーションフラグ = true
    - Contact Lens のセンチメント = NEGATIVE かつ確信度 ≥ 0.7
And   抽出件数・コンタクト ID リストを CloudWatch Logs に記録する
```

**補足チェックリスト**:

- [ ] 対象期間の低品質コンタクト数が 0 件の場合は処理をスキップし「改善提案なし」を記録
- [ ] Contact Lens API レート制限対策: 指数バックオフリトライ（最大 3 回）

---

### US-3.2: ナレッジギャップ分析

**ペルソナ**: アキコ（コンテンツ管理者）
**ストーリー**:
> コンテンツ管理者として、どのトピックの情報がわかりにくいかを AI が自動分析してくれるので、改善の焦点を絞れる。

**受け入れ基準（BDD）**:

```gherkin
Given  低品質コンタクトのリストが取得されている
When   分析 Lambda が各コンタクトの会話テキストを Claude に送信する
Then   Claude がトピックカテゴリを分類する（例：「振込手数料」「住宅ローン」「口座開設」）
And   同一カテゴリが複数コンタクトで検出された場合は件数を集計する
And   「わかりにくさスコア」（低スコア頻度 × エスカレーション率）を算出する
```

**補足チェックリスト**:

- [ ] Claude へ送信するテキストは PII マスク済みのサマリーのみ（生会話テキスト不使用）
- [ ] カテゴリ分類プロンプトは Secrets Manager で管理（バージョン管理可能）

---

### US-3.3: 週次改善提案生成

**ペルソナ**: アキコ（コンテンツ管理者）
**ストーリー**:
> コンテンツ管理者として、AI が毎週わかりにくい箇所の改善案を最大 10 件提示してくれるので、データに基づいた FAQ 改善ができる。

**受け入れ基準（BDD）**:

```gherkin
Given  ナレッジギャップ分析が完了している
When   Claude がわかりにくさスコア上位を元に改善提案を生成する
Then   最大 10 件の提案を DynamoDB の提案テーブルへ書き込む
And   各提案には「提案日時・優先度スコア・対象 URL・改善案テキスト（200 字以内）・ステータス=pending」が含まれる

Given  前週に「承認」ステータスとなった提案が存在する
When   今週の分析時に同一 URL の改善提案が再生成されそうになる
Then   未対応（pending）の既存提案があれば重複生成をスキップする
```

**補足チェックリスト**:

- [ ] 提案テーブルの TTL: 90 日（対応有無にかかわらず自動削除）
- [ ] 生成された提案数が 0 件の場合はダッシュボードに「今週の改善提案はありません」を表示

---

## Epic 4: オムニチャネルセッション継続

> **エピック概要**: 音声↔チャット間のチャネル切り替え時に会話コンテキストを引き継ぎ、シームレスな体験を提供する。

---

### US-4.1: 音声からチャットへの切り替え

**ペルソナ**: ユキ（顧客）
**ストーリー**:
> 顧客として、電話中に「チャットに切り替えたい」と言うとそのまま会話の続きをチャットで行えるので、声を出しにくい状況でも継続して問い合わせできる。

**受け入れ基準（BDD）**:

```gherkin
Given  顧客が音声コンタクト中に「チャットに切り替えてください」と発話する
When   Lex v2 がチャネル切り替えインテントを検出する
Then   「アプリのチャット画面へご案内します。会話の内容を引き継ぎます」とアナウンスする
And   現在のコンタクト ID をアプリへ Push 通知で送信する
And   音声コンタクトを一時停止（または終了）する

Given  顧客がアプリのチャット画面を開く
When   コンタクト ID をキーに Connect Chat セッションを開始する
Then   直近 5 ターンの会話サマリーがチャット画面の冒頭に表示される
And   顧客は前の会話の続きからテキストで問い合わせを継続できる
```

**補足チェックリスト**:

- [ ] Push 通知送信失敗時は SMS またはアプリバッジで代替案内
- [ ] 切り替え後 5 分以内にチャットが開始されない場合はセッションを破棄

---

### US-4.2: チャットから音声への切り替え

**ペルソナ**: ユキ（顧客）
**ストーリー**:
> 顧客として、チャット中に「電話で話したい」と入力すると、AI との会話を引き継いだままオペレーターまたは AI 音声に切り替えられる。

**受け入れ基準（BDD）**:

```gherkin
Given  顧客がチャット中に「電話で話したい」「電話してください」と入力する
When   Lex v2 がチャネル切り替えインテントを検出する
Then   「折り返し電話番号を入力してください」とチャットで返答する

Given  顧客が電話番号を入力する
When   Lambda が Amazon Connect の発信コンタクトフローをトリガーする
Then   登録番号へ自動発信し接続時に現在の会話サマリーをフローに引き渡す
```

**補足チェックリスト**:

- [ ] 電話番号のバリデーション: 日本の固定・携帯番号フォーマット（正規表現チェック）
- [ ] 発信前に「○○-○○○○-○○○○ へ電話します。よろしいですか？」確認メッセージを表示

---

### US-4.3: 有人対応後の AI 復帰と CSAT 誘導

**ペルソナ**: ユキ（顧客）、ケンジ（オペレーター）
**ストーリー**:
> オペレーターとして、対応完了後に AI へ制御を戻すだけで自動的に CSAT アンケートが実施されるので、手動でアンケートを案内する手間が省ける。

**受け入れ基準（BDD）**:

```gherkin
Given  オペレーターがコンタクトを解決し「AI へ転送」ボタンを押す
When   Connect フローが AI エージェントフローへリルーティングする
Then   AI が「担当者のご対応はいかがでしたか？」と CSAT アンケートを開始する（US-1.4 参照）
And   オペレーター ID とコンタクト ID を CSAT レコードに紐づけて保存する
```

**補足チェックリスト**:

- [ ] オペレーターが転送ボタンを押さずに切断した場合は CSAT をスキップしてセッション終了
- [ ] AI 復帰後の CSAT は最大 3 ターンで完了できる設計

---

## Epic 5: SDK ネイティブアプリ組み込み

> **エピック概要**: Amazon Connect Chat SDK を使用してネイティブアプリにチャット機能を組み込み、顧客プロファイル連携を実現する。

---

### US-5.1: ネイティブアプリからのチャット開始

**ペルソナ**: ユキ（顧客）
**ストーリー**:
> アプリユーザーとして、アプリ内の「お問い合わせ」ボタンをタップするだけでチャットが始まるので、ブラウザへ遷移したり電話番号を調べたりせずに問い合わせできる。

**受け入れ基準（BDD）**:

```gherkin
Given  顧客がアプリの「お問い合わせ」ボタンをタップする
When   Connect Chat SDK（iOS: Swift / Android: Kotlin）が StartChatContact API を呼び出す
Then   新規チャットセッションが確立されウェルカムメッセージが表示される
And   顧客は 5 秒以内にチャット画面が表示される

Given  ネットワーク接続が切れてから復旧する
When   SDK が再接続を検出する
Then   既存セッションへ自動再接続し会話履歴が復元される
```

**補足チェックリスト**:

- [ ] SDK バージョン: Amazon Connect Chat SDK v3 以上
- [ ] セッション確立に 5 秒以上かかる場合はスピナーとタイムアウトメッセージを表示
- [ ] オフライン時はチャット開始不可として「通信環境をご確認ください」を表示

---

### US-5.2: 顧客プロファイル属性付与

**ペルソナ**: ユキ（顧客）、タロウ（IT）
**ストーリー**:
> システム管理者として、顧客の au ID ハッシュがコンタクトに自動的に紐づくので、個人情報を平文で扱わずにパーソナライズ機能が実現できる。

**受け入れ基準（BDD）**:

```gherkin
Given  顧客がアプリ内にログイン済み（au ID セッション有効）
When   Chat SDK が StartChatContact を呼び出す
Then   アプリが au ID を SHA-256 でハッシュ化し customerId 属性として Connect へ渡す

Given  Lambda RAG ハンドラーがコンタクトを処理する
When   コンタクト属性から customerId を取得する
Then   DynamoDB の顧客履歴テーブルへ customerId をキーに顧客プロファイルを参照する
And   顧客プロファイルを Bedrock プロンプトに注入してパーソナライズ回答を生成する
```

**補足チェックリスト**:

- [ ] 未ログイン（au ID なし）の場合は customerId = "anonymous" でセッション開始
- [ ] ハッシュ化は必ずアプリ側（クライアント）で実行し、サーバーへ平文 au ID は送信しない

---

## Epic 6: 顧客履歴管理とパーソナライズ

> **エピック概要**: 顧客ごとの会話サマリーを 3 ヶ月保持し、次回コンタクト時にパーソナライズされた回答を提供する。

---

### US-6.1: 会話サマリーの自動生成と保存

**ペルソナ**: ユキ（顧客）
**ストーリー**:
> 顧客として、過去に問い合わせた内容を AI が覚えているので、次回同じことを一から説明しなくて済む。

**受け入れ基準（BDD）**:

```gherkin
Given  コンタクトが終了する（切断または CSAT 完了）
When   後処理 Lambda が Connect の ContactEnded イベントを受信する
Then   全会話ターンを Claude で 100 字以内に要約する
And   Amazon Comprehend で PII を検出しマスク処理する（氏名→[NAME]、口座番号→[ACCOUNT]）
And   DynamoDB 履歴テーブルへ {customerId, contactId, summary, channel, timestamp, ttl} を書き込む
And   TTL = 現在時刻 + 90 日 を設定する
```

**補足チェックリスト**:

- [ ] anonymous ユーザーの会話は保存しない
- [ ] サマリー生成エラー時は空サマリーで保存（次回コンタクトへの影響を最小化）
- [ ] 保存完了後、CRM API エンドポイントへ同一サマリーを非同期 POST する（スコープ外 API の呼び出し口のみ実装）

---

### US-6.2: パーソナライズされた回答提供

**ペルソナ**: ユキ（顧客）
**ストーリー**:
> リピーターの顧客として、過去の問い合わせ内容に基づいた的確なアドバイスをもらえるので、毎回一般的な説明を受けるより時間が節約できる。

**受け入れ基準（BDD）**:

```gherkin
Given  customerId が有効な顧客が新規コンタクトを開始する
When   Lambda RAG ハンドラーがコンタクトを処理する
Then   DynamoDB から直近 5 件の会話サマリーを取得する（timestamp 降順）
And   サマリーを RAG プロンプトの「過去の会話」セクションへ注入する
And   Claude が過去履歴を踏まえた回答を生成する

Given  顧客の履歴が存在しない（初回または 90 日超経過）
When   DynamoDB クエリの結果が空
Then   過去履歴なしとして通常の RAG 回答を提供する（エラーにしない）
```

**補足チェックリスト**:

- [ ] 履歴取得の DynamoDB クエリは P99 ≤ 100ms を目標（プロジェクション指定で最小フィールドのみ取得）
- [ ] プロンプト内の履歴セクションは最大 500 トークンに収まるよう切り詰め処理を行う

---

### US-6.3: CRM への会話サマリー書き込み

**ペルソナ**: ケンジ（オペレーター）
**ストーリー**:
> オペレーターとして、AI 対応の会話内容が自動的に CRM に記録されるので、顧客の問い合わせ履歴を CRM で一元管理できる。

**受け入れ基準（BDD）**:

```gherkin
Given  US-6.1 で DynamoDB への保存が完了する
When   CRM 書き込み Lambda が非同期で起動する
Then   設定された CRM API エンドポイントへ {customerId, contactId, summary, channel, timestamp} を POST する
And   HTTP 200 レスポンスを受信した場合は書き込み成功としてログを記録する
And   HTTP 4xx/5xx の場合は SQS デッドレターキューへメッセージを送信しアラームを発報する
```

**補足チェックリスト**:

- [ ] CRM API エンドポイントと認証情報は Secrets Manager で管理
- [ ] anonymous ユーザーは CRM 書き込みをスキップ
- [ ] リトライ: 指数バックオフ 最大 3 回（2s → 4s → 8s）

---

## Epic 7: 管理ダッシュボード

> **エピック概要**: Amplify + React で構築した管理ダッシュボードで、週次改善提案の確認・承認と利用統計の閲覧を行う。

---

### US-7.1: 週次改善提案の確認と承認

**ペルソナ**: アキコ（コンテンツ管理者）
**ストーリー**:
> コンテンツ管理者として、ダッシュボードで毎週の改善提案一覧を確認し承認・却下できるので、データに基づいたコンテンツ改善サイクルを回せる。

**受け入れ基準（BDD）**:

```gherkin
Given  アキコがダッシュボードにログインする
When   「改善提案」タブを開く
Then   今週生成された提案が優先度スコア降順で最大 10 件表示される
And   各提案には「対象 URL・改善案テキスト・優先度スコア・ステータス（pending/approved/rejected/hold）」が表示される

Given  アキコが提案の「承認」ボタンをクリックする
When   API バックエンド（Lambda）が DynamoDB 提案テーブルを更新する
Then   ステータスが "approved" に変わり、更新日時が記録される
And   画面上でリアルタイムにステータスが反映される

Given  アキコが提案の「却下」ボタンをクリックする
When   却下理由（任意テキスト入力）を入力して確定する
Then   ステータスが "rejected" に変わり、却下理由が記録される
```

**補足チェックリスト**:

- [ ] 改善提案一覧は週を選択して過去分も参照できる（最大 12 週分）
- [ ] ページング: 1 ページ 10 件表示、前後ナビゲーション付き
- [ ] CSV エクスポートボタンで提案一覧をダウンロード可能

---

### US-7.2: 利用統計ダッシュボードの閲覧

**ペルソナ**: アキコ（コンテンツ管理者）、タロウ（IT）
**ストーリー**:
> 管理者として、週次・月次の利用統計をダッシュボードで可視化できるので、AI エージェントのパフォーマンスを継続的にモニタリングできる。

**受け入れ基準（BDD）**:

```gherkin
Given  管理者がダッシュボードの「統計」タブを開く
When   デフォルトで直近 7 日間のデータが表示される
Then   以下の指標がグラフ・数値で表示される
    - コンタクト件数（音声/チャット別）
    - エスカレーション率（%）
    - 平均 CSAT スコア
    - 平均会話ターン数
    - AI 解決率（エスカレーションなしで終了した割合）

Given  管理者が期間を「直近 30 日」に変更する
When   期間選択 UI を操作する
Then   表示データが選択期間に切り替わる（3 秒以内）
```

**補足チェックリスト**:

- [ ] データソース: DynamoDB + CloudWatch Metrics を Lambda API で集計
- [ ] グラフライブラリ: Recharts（Amplify / React エコシステムと親和性が高く軽量）
- [ ] モバイルブラウザでも閲覧可能なレスポンシブデザイン

---

### US-7.3: ダッシュボードへの認証アクセス

**ペルソナ**: アキコ（コンテンツ管理者）、タロウ（IT）
**ストーリー**:
> システム管理者として、ダッシュボードへのアクセスを Cognito で制御できるので、権限のないユーザーが改善提案や統計データを閲覧・操作できない。

**受け入れ基準（BDD）**:

```gherkin
Given  未認証ユーザーがダッシュボード URL にアクセスする
When   Amplify が認証状態を確認する
Then   Cognito ホステッド UI のログイン画面にリダイレクトされる

Given  管理者がメールアドレスとパスワードで認証する
When   Cognito が認証成功を返す
Then   ダッシュボードのトップ画面にリダイレクトされアクセストークンが Cookie に保存される

Given  アクセストークンの有効期限（1 時間）が切れる
When   管理者がページ操作を行う
Then   リフレッシュトークンで自動更新されセッションが継続する
And   リフレッシュトークンも期限切れ（30 日）の場合は再ログイン画面へ遷移する
```

**補足チェックリスト**:

- [ ] MFA（TOTP）をオプションとして有効化
- [ ] ログイン失敗 5 回でアカウント一時ロック（Cognito デフォルト設定）
- [ ] パスワードポリシー: 最低 8 文字、大文字・小文字・数字・記号を含む
