# Requirements Document

# au Jibun Bank AI Agent

## Intent Analysis Summary

| 項目 | 内容 |
|---|---|
| **ユーザーリクエスト** | Amazon Connect を基盤とした音声・テキスト対応の生成 AI カスタマーサポートエージェントの構築。au じぶん銀行公式サイト・FAQ をナレッジソースとし、自己改善サイクル・オムニチャネル・顧客履歴パーソナライズ・週次改善提案を実現する。 |
| **リクエスト種別** | 新規プロジェクト（グリーンフィールド） |
| **スコープ見積もり** | システム全体（複数コンポーネント・複数 AWS サービスにまたがる） |
| **複雑度見積もり** | 高（生成 AI・コンタクトセンター・ウェブクローリング・パーソナライゼーション・管理 UI が連動） |

## 1. 機能要件

### FR-01: AI 音声・テキストエージェント

| ID | 要件 |
|---|---|
| FR-01-01 | Amazon Connect インバウンド電話コールを受け付け、Amazon Lex v2（日本語）で自然言語理解を行う |
| FR-01-02 | Amazon Connect チャットを受け付け、同一 Lex v2 ボットでテキスト対話を処理する |
| FR-01-03 | 音声 ASR は Amazon Lex v2 の組み込み音声認識（Amazon Transcribe バックエンド）を使用する |
| FR-01-04 | 音声合成（TTS）は Amazon Polly Neural「Kazuha」(ja-JP) を使用する（最も自然な AWS ネイティブ音声） |
| FR-01-05 | カスタム RAG（DynamoDB ベクトルストア + Lambda コサイン類似度検索 + Bedrock Claude）でナレッジベースから回答を生成する |
| FR-01-06 | 回答生成に使用した出典ページ（URL）をレスポンスに付与する |
| FR-01-07 | ナレッジベースに回答が見つからない場合は有人オペレーターへエスカレーションする |

### FR-02: 自動ナレッジクローリング＆更新パイプライン

| ID | 要件 |
|---|---|
| FR-02-01 | `https://www.au-jibunbank.co.jp/` 配下のページおよび FAQ セクションを **週次（毎週日曜深夜）** で自動クローリングする |
| FR-02-02 | クローリング前に `robots.txt` を取得し、`Disallow` ルールを遵守する |
| FR-02-03 | 前回クローリング結果との差分を検出し、変更・追加ページのみ S3 へ保存する |
| FR-02-04 | 削除されたページは S3 および DynamoDB ベクトルストアから除外する（陳腐化防止） |
| FR-02-05 | クローリング完了後に差分チャンクの埋め込みベクトルを Amazon Titan Embeddings v2 で生成し DynamoDB へ書き込む |
| FR-02-06 | クローリング結果（件数・差分・エラー）を CloudWatch Logs へ出力する |
| FR-02-07 | 将来的な多言語対応を見越し、言語識別フィールドをメタデータに含める設計とする |

### FR-03: 自己改善サイクル

| ID | 要件 |
|---|---|
| FR-03-01 | Amazon Connect Contact Lens でコンタクト品質スコア・感情分析・エスカレーションフラグを取得する |
| FR-03-02 | 低スコア（閾値：CSAT < 3 / エスカレーション発生）のコンタクトを週次で抽出する |
| FR-03-03 | Amazon Bedrock を用いて抽出コンタクトのナレッジギャップを分析し、FAQ・ページのどのカテゴリが不足・不明瞭かを分類する |
| FR-03-04 | ナレッジギャップを「わかりにくさスコア」（低スコア頻度 × エスカレーション率）で順位付けし、上位 10 件を改善提案として生成する |
| FR-03-05 | 生成した改善提案を DynamoDB の提案テーブルへ書き込む（提案日時・優先度スコア・対象 URL・改善案テキスト） |

### FR-04: オムニチャネルセッション継続

| ID | 要件 |
|---|---|
| FR-04-01 | Amazon Connect のオムニチャネル機能（Unified Agent Experience）を使用して音声・チャットを統合管理する |
| FR-04-02 | 音声からチャットへ（またはその逆）のチャネル切り替え時に、同一コンタクト ID をセッションキーとして文脈を引き継ぐ |
| FR-04-03 | チャネル切り替え時に直近 N ターン（デフォルト: 5）の要約をプロンプトへ注入し、会話の連続性を担保する |
| FR-04-04 | 有人対応完了後、AI エージェントへ復帰し CSAT アンケートおよび終了フローを処理する |

### FR-05: SDK ネイティブアプリ組み込み

| ID | 要件 |
|---|---|
| FR-05-01 | Amazon Connect Chat SDK（iOS: Swift / Android: Kotlin）を使用してネイティブアプリへ組み込む |
| FR-05-02 | アプリから Chat SDK でコンタクト開始時、Amazon Connect 顧客プロファイルを使用（匿名開始後 Lambda で顧客属性付与） |
| FR-05-03 | 顧客識別に用いる Customer ID は au ID のハッシュ値（SHA-256）を使用し、平文 PII を Connect に渡さない |
| FR-05-04 | SDK 統合手順・設定値（インスタンスエンドポイント等）をドキュメントとして提供する |

### FR-06: 顧客履歴管理とパーソナライズ

| ID | 要件 |
|---|---|
| FR-06-01 | コンタクト終了時に Lambda が会話サマリー（Bedrock による要約）を生成し DynamoDB へ保存する |
| FR-06-02 | DynamoDB の顧客履歴は **3 ヶ月間** 保持し、TTL 設定で自動削除する |
| FR-06-03 | コンタクト終了時に CRM API へ会話サマリーを書き込む（CRM 連携は本スコープ外；Lambda から呼び出す API エンドポイントと認証方式は別途提供） |
| FR-06-04 | 次回コンタクト時に直近 5 件の履歴サマリーを RAG プロンプトへ注入し、パーソナライズされた回答を生成する |
| FR-06-05 | 会話テキスト内の PII（氏名・口座番号等）は Amazon Comprehend で検出しマスク処理してから保存する |

### FR-07: 管理ダッシュボード（Amplify + React）

| ID | 要件 |
|---|---|
| FR-07-01 | AWS Amplify + React で管理ダッシュボードを構築する（ランニングコスト最小化） |
| FR-07-02 | ダッシュボードに週次改善提案一覧（優先度スコア・対象 URL・提案内容・ステータス）を表示する |
| FR-07-03 | 管理者が各提案に対して「承認」「却下」「保留」のステータスを記録できる |
| FR-07-04 | ダッシュボードに利用統計（週次コンタクト件数・チャネル別割合・エスカレーション率・平均会話ターン数）を表示する |
| FR-07-05 | ダッシュボードへのアクセスは Amazon Cognito で認証する |
| FR-07-06 | 通知機能は実装しない（管理者がダッシュボードを確認する運用） |

## 2. 非機能要件（NFR）

### NFR-01: 可用性・パフォーマンス

| ID | 要件 |
|---|---|
| NFR-01-01 | Lambda ハンドラーのレスポンスタイムは Amazon Connect 制約（8 秒以内）を遵守する |
| NFR-01-02 | コンタクトフロー内 Lambda は P99 ≤ 5 秒を目標とする |
| NFR-01-03 | ナレッジベース RAG 回答生成は P99 ≤ 7 秒を目標とする |
| NFR-01-04 | 週次クローラーおよび改善提案生成は非同期実行（EventBridge Scheduler）のため応答タイム制約なし |
| NFR-01-05 | 同時コンタクト数は最大 100 セッションを想定し、Lambda の予約同時実行数を設定する |

### NFR-02: セキュリティ（Security Extension 適用）

| ID | 要件 |
|---|---|
| NFR-02-01 | DynamoDB・S3 バケット・CloudWatch Logs は保存時暗号化を有効化する（SECURITY-01） |
| NFR-02-02 | すべてのデータ転送は TLS 1.2 以上を強制する（SECURITY-01） |
| NFR-02-03 | API Gateway ステージには実行ログ・アクセスログを CloudWatch へ出力する（SECURITY-02） |
| NFR-02-04 | 全 Lambda・Amplify バックエンドに構造化ログ（JSON）を実装し、PII をログに出力しない（SECURITY-03） |
| NFR-02-05 | Lambda 実行ロールは最小権限（リソース ARN 指定）で設計する |
| NFR-02-06 | Secrets Manager でクレデンシャルを管理し、環境変数への平文記録を禁止する |
| NFR-02-07 | PII は Amazon Comprehend でマスク処理してから DynamoDB に保存する |
| NFR-02-08 | S3 バケットはパブリックアクセスブロックを有効化する |

### NFR-03: スケーラビリティ

| ID | 要件 |
|---|---|
| NFR-03-01 | DynamoDB ベクトルストアは大規模コンテンツ（ページ 500+・FAQ 500+、約 3,000 チャンク、~20MB ベクトルデータ）を収容できる設計とする |
| NFR-03-02 | DynamoDB はオンデマンドキャパシティモードを使用し、スパイク時の自動スケーリングに対応する |
| NFR-03-03 | 週次クローラーは対象サイトへの負荷を考慮し、リクエスト間隔に 1〜3 秒のランダムディレイを設ける |

### NFR-04: 保守性・拡張性

| ID | 要件 |
|---|---|
| NFR-04-01 | インフラは AWS CDK v2（TypeScript）で IaC 管理し、環境（dev/staging/prod）を CDK context で切り替える |
| NFR-04-02 | Lambda は Python 3.12、依存管理は uv + pyproject.toml を使用する |
| NFR-04-03 | 多言語対応の将来拡張を考慮し、言語設定を定数・環境変数で切り替え可能な設計とする（MVP は日本語のみ） |
| NFR-04-04 | GitHub Actions で lint（ruff + mypy）・テスト（pytest + moto）・CDK synth を自動実行する |

### NFR-05: テスト（Property-Based Testing Extension 適用）

| ID | 要件 |
|---|---|
| NFR-05-01 | pytest + pytest-asyncio + moto でユニットテストを実装し、内部コードパスのカバレッジ 80% 以上を目標とする |
| NFR-05-02 | ビジネスロジック・データ変換・シリアライゼーション処理にはプロパティベーステスト（hypothesis）を適用する（PBT-01） |
| NFR-05-03 | PII マスキング・会話サマリー生成・差分検出ロジックには round-trip・invariant プロパティを定義する |
| NFR-05-04 | 統合テストは LocalStack または dev 環境の実 AWS リソースで実行する |

## 3. 制約事項

| 制約 | 内容 |
|---|---|
| AWS リージョン | `ap-northeast-1`（東京）のみ使用 |
| クローリング対象 | `https://www.au-jibunbank.co.jp/` 配下のみ（他ドメイン禁止） |
| robots.txt 遵守 | クローリング前に必ず確認し、Disallow ルールを厳守 |
| CRM 連携 | API エンドポイント・認証方式は外部提供のためスコープ外（呼び出し口のみ実装） |
| Connect インスタンス | 新規作成（CDK で完全プロビジョニング） |
| 言語 | 日本語のみ（MVP） |
| トランザクション処理 | 口座取引・残高照会等の API 連携はスコープ外 |
| データ保持期間 | DynamoDB は 3 ヶ月（TTL）、長期保管は CRM 側 |

## 4. 主要な技術決定事項

| 決定 | 内容 | 根拠 |
|---|---|---|
| 音声 ASR | Amazon Lex v2 組み込み（Amazon Transcribe バックエンド） | AWS ネイティブ・追加 API キー不要 |
| 音声 TTS | Amazon Polly Neural「Kazuha」(ja-JP) | 最も自然な AWS ネイティブ音声・外部依存なし |
| ナレッジ生成 AI | Amazon Bedrock Claude claude-sonnet-4-6 | 最新・高品質な日本語対応 |
| ベクトルストア | DynamoDB + Lambda カスタム RAG | 外部サービス不要・AWS ネイティブ・月額コスト最小 |
| 埋め込み生成 | Amazon Titan Embeddings v2（Bedrock） | AWS ネイティブ・コスト効率 |
| 顧客履歴 DB | Amazon DynamoDB（オンデマンド） | サーバーレス・スパイク対応 |
| ダッシュボード | AWS Amplify + React | ランニングコスト最小・カスタム UI |
| クローリング頻度 | 週次（毎週日曜深夜） | サーバー負荷最小・FAQ 更新サイクルと一致 |
| 顧客認証 | Connect 顧客プロファイル（匿名開始 → Lambda で属性付与） | ユーザー指定 |

## 5. コスト見積もり

**前提条件**: 月間 100 セッション未満、音声 50 件×平均 5 分・チャット 50 件×10 メッセージ、$1 = 150 円

### 月次ランニングコスト内訳

| サービス | 想定使用量 | 月額（USD） | 月額（円） |
|---|---|---|---|
| Amazon Connect（音声） | 250 分 × $0.018/分 | $4.50 | 675 円 |
| Amazon Connect（チャット） | 500 メッセージ × $0.004 | $2.00 | 300 円 |
| Contact Lens（会話分析） | 250 分 × $0.015/分 | $3.75 | 563 円 |
| Amazon Lex v2（音声+テキスト） | 1,000 リクエスト × $0.004 | $2.00 | 300 円 |
| Amazon Polly Neural TTS | 25,000 文字 × $0.000016 | $0.40 | 60 円 |
| Bedrock Claude（RAG 回答生成） | 入力 1M token + 出力 250K token | $6.75 | 1,013 円 |
| Titan Embeddings v2（週次差分） | ~100 チャンク × 300 tokens | $0.01 | 2 円 |
| DynamoDB（ベクトルストア+履歴） | オンデマンド、~20MB | $0.50 | 75 円 |
| S3（クローリングコンテンツ） | ~10 GB × $0.025/GB | $0.25 | 38 円 |
| AWS Amplify（ダッシュボード） | ホスティング + ビルド | $1.00 | 150 円 |
| Amazon Cognito | <50K MAU | $0.00 | 0 円 |
| Amazon Comprehend（PII 検出） | ~100K ユニット | $0.10 | 15 円 |

| Lambda | ~10K 実行 | $0.00 | 0 円 |
| **合計** | | **~$21.76** | **~3,266 円** |

### 目標対比

| 項目 | 値 |
|---|---|
| 月額目標 | 5,000 円以下 |
| 月額見積もり | **約 3,266 円** |
| 目標との差 | **約 1,734 円の余裕（目標比 65%）** |

### コストボトルネック解消済み

| ボトルネック | 旧設計 | 新設計（B-2 案） | 削減効果 |
|---|---|---|---|
| ベクトルストア | OpenSearch Serverless ~$700/月 | DynamoDB カスタム RAG ~$0.50/月 | **▲$699.50（約 105,000 円削減）** |
| 外部 API | Pinecone（外部サービス） | なし（完全 AWS ネイティブ） | 管理コスト削減 |

### コスト超過リスク

| リスク | 条件 | 対策 |
|---|---|---|
| Bedrock Claude 費用急増 | セッション数が 100 件を大幅超過 | CloudWatch アラーム + Lambda スロットリング |
| Contact Lens 費用増 | 長時間通話が増加 | 通話時間上限フロー設定 |
| DynamoDB スキャン費用 | ベクトル全件スキャンが高頻度 | Lambda 内キャッシュ（/tmp 最大 10 GB）で頻度抑制 |

## 6. スコープ外（明示的除外）

- 口座残高照会・振込処理等のトランザクション API 連携
- 多言語対応の実装（拡張ポイントの設計のみ含む）
- ウェブサイトへのライブチャットウィジェット組み込み
- 改善提案の通知機能（Slack・メール）
- CRM システムの実装・管理（API 呼び出し口のみ提供）
- 他行・他ドメインのクローリング

---

## 6. 前提条件

- AWS アカウントが存在し、ap-northeast-1 で各サービスが利用可能であること
- Amazon Connect インスタンスは CDK で新規作成する
- Amazon Bedrock の Claude claude-sonnet-4-6 および Titan Embeddings v2 モデルへのアクセスが事前に有効化されていること
- CRM API のエンドポイント・認証方式は別途提供されること
- Cognito ユーザープールの管理者アカウントが別途発行されること
- 外部サービス（Pinecone 等）のアカウントは不要（完全 AWS ネイティブ）
